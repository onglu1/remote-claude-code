## 风险点
- tick 放置顺序：HUD 解析必须在 ask 早返回**之前**执行，否则待答期 HUD 不更新。但 HUD 广播不能干扰 ask 的 return（先广播 HUD 再走 ask 逻辑）。
- 去重抖动：百分比每分钟才变，签名稳定；但 raw 含进度条字符会随 % 变 → 用 JSON.stringify(hud) 整体签名即可，% 变才重发（符合预期）。
- 解析鲁棒：只认 trim 后以 `[` 开头的行为 HUD 首行；找不到 → null（无 HUD/未配 statusLine 的会话不显示，正常）。
- 前端样式落点：先确认聊天相关 CSS 在哪个文件，避免新建无谓文件。
